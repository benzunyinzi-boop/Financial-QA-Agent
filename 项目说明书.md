# LangChain ReAct Agent 智能客服项目说明书

## 1. 项目概述

### 1.1 项目名称
LangChain ReAct Agent 智能客服（扫地机器人垂直领域）

### 1.2 项目目标
构建一个面向扫地机器人售前/售后场景的 AI 智能客服系统，支持：
- 常见问题问答
- 故障排查与处理建议
- 基于知识库的检索增强回答（RAG）
- 可扩展的工具调用与多轮对话能力

### 1.3 业务价值
- 降低人工客服重复问答成本
- 提升用户自助解决问题效率
- 通过结构化知识库沉淀提升回答一致性
- 为后续运营分析和自动化服务打基础

## 2. 当前系统形态与演进方向

### 2.1 当前形态（截至 2026-05-22）
项目已具备两条可运行路径：
- Streamlit 快速体验入口：`app.py`
- FastAPI + React 前后端分离：`api/` + `frontend/`

当前建议将 FastAPI + React 作为主线交付形态，Streamlit 作为调试/演示入口。

### 2.2 目标形态
遵循四层架构：`api -> agent -> rag -> model`，实现职责清晰、可测试、可扩展的工程化 AI 应用。

## 3. 技术选型

- 后端语言：Python 3.10+
- Web 框架：FastAPI（主线）/ Streamlit（体验入口）
- AI 框架：LangChain 0.3.x + LangGraph 0.2.x
- 大模型：DashScope `qwen3-max`（可按任务切换 `qwen-plus` / `qwen-turbo`）
- Embedding：DashScope `text-embedding-v4`
- 向量库：Chroma 0.5.x（开发/测试），Milvus（生产规划）
- 前端：React 18 + TypeScript + TailwindCSS v4 + shadcn/ui
- 状态管理：Zustand
- 配置：YAML + python-dotenv
- 日志：Python logging（文件输出 + 分级）
- 部署：Docker + docker-compose

## 4. 分层架构设计

### 4.1 分层职责

1. api 层（`api/`）
- 负责路由、参数校验、协议适配（HTTP/SSE/WebSocket）
- 不承载核心业务逻辑

2. agent 层（`agent/`）
- 负责 ReAct 推理流程、工具编排、对话状态管理
- 调度 rag 层能力，不直接处理存储细节

3. rag 层（`rag/`）
- 负责文档加载、分块、向量化、召回、上下文构建
- 仅依赖 model 层提供的模型能力

4. model 层（`model/`）
- 统一封装 LLM 与 Embedding 实例创建
- 负责模型参数、超时、重试、降级策略

### 4.2 依赖约束
- 允许：`api -> agent -> rag -> model`
- 禁止：跨层反向调用、循环依赖
- 禁止：api 直接调用 rag/model

## 5. 核心功能说明

### 5.1 智能对话
- 支持多轮上下文记忆
- 支持流式输出（SSE）
- 支持工具增强回答

### 5.2 RAG 检索增强
- 知识文档向量化入库
- Top-K 召回（当前主策略）
- 召回上下文拼接后交由 LLM 生成答案
- 检索失败时可降级到通用回答

### 5.3 知识库管理
- 支持文档导入（TXT/PDF 等）
- 支持文档列表与检索调用
- 支持后续扩展：文档更新、重建索引、版本控制

### 5.4 可观测性
- 分级日志（DEBUG/INFO/WARNING/ERROR）
- 关键链路记录（请求、检索、模型调用、异常）
- 为 trace_id 全链路跟踪预留上下文

## 6. 目录说明（与当前仓库一致）

- `app.py`：Streamlit 入口
- `start_api.py`：FastAPI 启动脚本（含环境变量检查）
- `api/`：路由、schema、数据库相关
- `agent/`：ReAct Agent 与工具中间件
- `rag/`：向量检索与 RAG 服务
- `model/`：模型工厂
- `prompts/`：提示词模板
- `config/`：YAML 配置
- `utils/`：日志、配置、路径、文件等通用能力
- `frontend/`：React 前端工程
- `deploy/`：部署脚本与部署文档
- `data/`：业务知识文档与外部数据

## 7. 接口与协议约定

### 7.1 REST 设计
- 路径风格：kebab-case
- 版本前缀：`/api/v1`
- 资源命名：复数优先

### 7.2 统一响应格式

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

### 7.3 流式输出规范
- SSE 数据格式：`data: {token}\n\n`
- 结束标记：`data: [DONE]\n\n`

## 8. 配置与环境变量

### 8.1 必需环境变量
- `DASHSCOPE_API_KEY`：DashScope 访问密钥（必填）

### 8.2 配置文件
- `config/agent.yml`：Agent 行为配置
- `config/rag.yml`：RAG 参数（如 top_k、chunk 配置）
- `config/chroma.yml`：向量库配置
- `config/prompts.yml`：提示词映射配置

### 8.3 安全要求
- 禁止在代码中硬编码密钥
- `.env` 不入库
- 日志中敏感信息必须脱敏

## 10. 质量保障

### 10.1 代码规范
- 遵循 PEP 8
- 命名统一（snake_case/PascalCase/UPPER_CASE）
- 避免裸 `except`

### 10.2 测试策略
- 使用 `pytest + pytest-asyncio`
- LLM 响应以 mock 为主，避免外部依赖不稳定
- 优先覆盖 Agent、RAG、API 核心流程

### 10.3 性能基线（目标）
- 首字延迟 p99 < 2s
- RAG 检索 p99 < 500ms
- LLM 调用超时默认 30s

## 11. 架构演进方案对比

### 方案 A：保持 Chroma + 单体服务（短中期推荐）
优点：
- 研发与部署成本低
- 适合中小规模知识库快速迭代
- 运维复杂度低

缺点：
- 大规模向量数据下扩展能力有限
- 多副本一致性与高可用能力有限

适用阶段：
- 验证期、业务快速迭代期

### 方案 B：迁移 Milvus + 服务化拆分（中长期推荐）
优点：
- 更强的向量检索扩展能力与性能上限
- 更适合多租户与高并发场景
- 易于建立独立检索服务

缺点：
- 部署与运维复杂度显著提高
- 初期开发联调成本更高

适用阶段：
- 生产规模化、SLA 要求较高阶段

结论：
- 当前建议优先采用方案 A，待知识量级与并发达到阈值后平滑迁移到方案 B。

## 12. 风险与治理

- Prompt 注入风险：需增加输入预处理和策略过滤
- 模型不可用风险：需实现主备模型降级和模板兜底
- 成本失控风险：需记录 token 消耗并设置预算告警
- 检索偏差风险：需引入 reranker 和离线评测机制

## 13. 里程碑建议

1. M1（当前）：打通 FastAPI + React + RAG 基础能力
2. M2：补齐测试体系与观测指标（trace_id、耗时、token）
3. M3：引入 reranker、缓存与降级策略
4. M4：评估并推进 Milvus 迁移与服务拆分
